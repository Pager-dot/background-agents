Automations define the repository, branch, model, and instructions once, then Open-Inspect starts a new session whenever the trigger fires.

## Trigger Types

| Type | Description | Status |
|------|-------------|--------|
| **Schedule** | Run on a cron schedule | Available |
| **Inbound Webhook** | Trigger from any system via HTTP POST | Available |
| **Sentry Alert** | Trigger from a Sentry Custom Integration | Available |
| **GitHub Event** | Trigger on GitHub activity | Planned |
| **Linear Event** | Trigger on Linear activity | Planned |

Common use cases: nightly dependency updates, deploy/incident response, Sentry triage, recurring report generation.

---

## Creating an Automation

Navigate to **Automations** in the sidebar, then click **Create Automation**.

### Required Fields

| Field | Description |
|-------|-------------|
| **Trigger Type** | Schedule, inbound webhook, or Sentry alert |
| **Name** | Short label (max 200 characters) — prefixed with `[Auto]` in session titles |
| **Repository** | GitHub repository to run against (cannot be changed after creation) |
| **Instructions** | Prompt sent to the agent each time the automation fires (max 10,000 characters) |

### Optional Fields

| Field | Description |
|-------|-------------|
| **Branch** | Base branch for each session (defaults to repo default branch) |
| **Model** | AI model to use (defaults to system default) |
| **Reasoning** | Reasoning level for models that support it |
| **Conditions** | Trigger filters for event-driven automations |

## Schedule Automations

### Schedule Presets

| Preset | Description | Controls |
|--------|-------------|---------|
| **Every hour** | Runs at the top of the hour | None |
| **Daily** | Once per day | Hour picker (12-hour AM/PM) |
| **Weekly** | Once per week | Day-of-week + hour |
| **Monthly** | Once per month | Day-of-month (1st–28th) + hour |
| **Custom** | Any valid 5-field cron | Text input with live validation |

### Custom Cron Expressions

Standard 5-field format: `minute hour day-of-month month day-of-week`

| Expression | Meaning |
|-----------|---------|
| `*/15 * * * *` | Every 15 minutes |
| `0 9 * * *` | Daily at 9:00 AM |
| `30 14 * * 1-5` | Weekdays at 2:30 PM |
| `0 0 1 * *` | First of every month at midnight |

<Warning>
  Minimum schedule interval is **15 minutes**. Expressions that fire more frequently are rejected. Six-field expressions (with seconds) are not supported.
</Warning>

## Inbound Webhooks

Use **Inbound Webhook** when you want any external system to trigger an automation with a JSON payload.

### Setup

1. Create an automation with **Trigger Type = Inbound Webhook**
2. Copy the generated webhook URL and API key shown after creation

<Note>
  The API key is only shown once — store it when you create the automation.
</Note>

### Request Requirements

| Requirement | Value |
|-------------|-------|
| Method | `POST` |
| Content-Type | `application/json` |
| Authentication | `Authorization: Bearer <api-key>` |
| Maximum payload size | 64 KB |

```bash
curl -X POST "https://<your-worker-url>/webhooks/automation/<automation-id>" \
  -H "Authorization: Bearer <api-key>" \
  -H "Content-Type: application/json" \
  -d '{"event":"deploy.failed","service":"api","environment":"prod"}'
```

### Response Format

```json
{ "ok": true, "triggered": 1, "skipped": 0 }
```

| Field | Meaning |
|-------|---------|
| `triggered` | Number of automation runs started |
| `skipped` | Number ignored due to duplicate delivery or concurrency protection |

### Error Responses

| Status | Meaning |
|--------|---------|
| `400` | Invalid JSON body |
| `401` | Missing or invalid API key |
| `404` | Automation not found or not a webhook automation |
| `413` | Payload too large |
| `415` | `Content-Type` was not `application/json` |

### What the Agent Receives

Open-Inspect prepends a context block to your instructions containing:
- The trigger type and time the webhook was received
- The JSON payload (truncated if necessary)
- A warning to treat the payload as untrusted input data

### Filtering with Conditions

Webhook automations support **JSONPath Filter** conditions. All filters must match for the automation to run.

Supported comparisons:

| Comparison | Meaning |
|-----------|---------|
| `eq` | Equal to |
| `neq` | Not equal to |
| `gt` | Greater than |
| `gte` | Greater than or equal to |
| `lt` | Less than |
| `lte` | Less than or equal to |
| `contains` | String contains substring |
| `exists` | Field is present |

Filter examples:

| Goal | Filter |
|------|--------|
| Run only for production | `$.environment eq "production"` |
| Run only for failed deploys | `$.status eq "failed"` |
| Run only when a field is present | `$.pull_request.number exists` |

<Note>
  Path syntax is limited to simple dot notation like `$.event.type`. Array indexing and recursive descent are not supported.
</Note>

### Idempotency

Include an `idempotencyKey` field in the JSON body to deduplicate retried deliveries. The same `idempotencyKey` will not create duplicate runs.

## Managing Automations

### Pause and Resume

**Pausing** stops the automation from firing — schedules won't run and event-driven automations ignore incoming events. **Resuming** reactivates the automation and resets the consecutive failure counter.

### Trigger Now

Click **Trigger Now** for a one-off run. Doesn't affect the next scheduled run time. Follows the same concurrency rules — rejected if a run is already active.

### Edit

You can change name, branch, model, instructions, schedule, and timezone at any time. Repository cannot be changed after creation.

### Delete

Deleting stops all future runs. Existing run history and sessions are preserved.

## Run History

Each automation's detail page shows chronological runs with status, duration, and session links.

### Run Statuses

| Status | Meaning |
|--------|---------|
| **Starting** | A session is being created |
| **Running** | Session is actively executing |
| **Completed** | Session finished successfully |
| **Failed** | Session encountered an error |
| **Skipped** | Previous run was still active (concurrency protection) |

---

## Automation Status

| Status | Meaning |
|--------|---------|
| **Enabled** | Running normally |
| **Degraded** | Enabled but has recent consecutive failures — failure count shown on badge |
| **Paused** | Not firing — manually or auto-paused |

---

## Concurrent Runs

For scheduled and manual triggers, only **one run per automation** can be active at a time. If a trigger fires while a previous run is still in progress, the new run is recorded as **Skipped**.

Event-driven automations use concurrency keys. Retries with the same `idempotencyKey` are treated as the same event, but separate deliveries without a shared key can overlap.

---

## Auto-Pause

If an automation fails **3 consecutive times**, it is automatically paused. Click **Resume** to re-enable and reset the failure counter.

<Note>
  Runs that time out (over 90 minutes) also count as failures toward the auto-pause threshold.
</Note>

---

## Limits

| Limit | Value |
|-------|-------|
| Automation name length | 200 characters |
| Instructions length | 10,000 characters |
| Minimum schedule interval | 15 minutes |
| Webhook payload size | 64 KB |
| Concurrent runs per automation | 1 (scheduled/manual triggers only) |
| Consecutive failures before auto-pause | 3 |
| Run execution timeout | 90 minutes |
