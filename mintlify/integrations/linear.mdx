The Linear bot integrates with Linear as a first-class **Linear Agent**. Users can `@mention` or assign the agent on any issue to trigger a background coding session. Progress is visible directly in Linear as native agent activities, and the resulting PR is linked back to the issue.

---

## How It Works

```
@OpenInspect on issue → Linear sends AgentSessionEvent webhook →
  Agent emits "Thinking..." → Resolves repo → Creates session →
  Agent emits "Working on owner/repo..." → Agent codes in sandbox →
  Completion callback → Agent emits "PR opened: <link>"
```

1. A user `@mentions` or assigns the agent on a Linear issue
2. Linear sends an `AgentSessionEvent` webhook to the bot worker
3. The worker emits a **Thought** activity visible in Linear as a thinking indicator
4. Resolves the target GitHub repo (see [Repo Resolution](#repo-resolution))
5. Creates an Open-Inspect coding session using the issue title and description as the prompt
6. Emits a **Response** activity with a link to the live session
7. When the agent completes, emits a final **Response** with the PR link

<Note>
  Follow-up messages on an issue with an active session are sent as additional prompts to the existing session — not new sessions. Stopping or cancelling the agent in Linear terminates the sandbox session.
</Note>

## Setup

### 1. Create a Linear OAuth Application

Go to **[Linear Settings → API → Applications → New](https://linear.app/settings/api/applications/new)**

Fill in:

| Field | Value |
|-------|-------|
| **Application name** | `OpenInspect` (this is how the bot appears in @mentions) |
| **Developer name** | Your org name |
| **Callback URL** | `https://<your-linear-bot-worker>/oauth/callback` |
| **Webhooks** | Enable, set URL to `https://<your-linear-bot-worker>/webhook` |
| **Webhook events** | Check **Agent session events**, **Issues**, **Comments** |
| **Public** | OFF (unless distributing to other workspaces) |

Note the **Client ID**, **Client Secret**, and **Webhook Signing Secret**.

---

### 2. Deploy via Terraform

Set `enable_linear_bot = true` and add to your `terraform.tfvars`:

```hcl
enable_linear_bot     = true
linear_client_id      = "your-client-id"
linear_client_secret  = "your-client-secret"
linear_webhook_secret = "your-webhook-signing-secret"
```

Then run:

```bash
terraform apply
```

The bot worker URL will be:

```
https://open-inspect-linear-bot-{deployment_name}.{subdomain}.workers.dev
```

---

### 3. Install the Agent in Your Workspace

Visit the OAuth authorization URL in your browser:

```
https://open-inspect-linear-bot-{deployment_name}.{subdomain}.workers.dev/oauth/authorize
```

This initiates the OAuth flow with `actor=app` and installs the agent in your Linear workspace.

<Warning>
  Installing the agent requires **admin permissions** in the Linear workspace.
</Warning>

After installation, `@OpenInspect` will appear in the mention and assignee menus for all issues.

---

### 4. Configure Repo Mapping (Optional)

The agent resolves repos automatically in most cases (see [Repo Resolution](#repo-resolution)). Static mappings are optional overrides for precise control.

All `/config/*` endpoints require an `Authorization` header with an HMAC-signed bearer token derived from `INTERNAL_CALLBACK_SECRET`.

#### Team → Repo Mapping

```bash
curl -X PUT https://<your-linear-bot-worker>/config/team-repos \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "YOUR_TEAM_ID": [
      { "owner": "your-org", "name": "frontend", "label": "frontend" },
      { "owner": "your-org", "name": "backend", "label": "backend" },
      { "owner": "your-org", "name": "main-repo" }
    ]
  }'
```

Each team maps to an array of repos. If a repo has a `label`, it only matches issues with that label. The first repo without a label is the default fallback.

#### Project → Repo Mapping

```bash
curl -X PUT https://<your-linear-bot-worker>/config/project-repos \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "LINEAR_PROJECT_ID": { "owner": "your-org", "name": "my-repo" }
  }'
```

Project mappings take the highest priority during repo resolution.

---

### 5. Configure Integration Settings (Optional)

In the Open-Inspect web UI, go to **Settings → Integrations → Linear** to configure:

- Default model and reasoning effort
- Whether users can override the model via preferences or issue labels
- Whether real-time tool progress activities are shown in Linear
- Which repos the Linear agent is enabled for (allowlist or all)

These settings can also be overridden per repository.

## Usage

On any Linear issue:

- Type `@OpenInspect` in a comment → agent picks up the issue
- Assign the issue to `OpenInspect` → agent picks it up
- Agent status is visible directly in Linear (thinking → working → done)

### Model Override via Labels

Add a `model:<name>` label to an issue to override the default model for that session:

| Label | Model |
|-------|-------|
| `model:opus` | Claude Opus |
| `model:sonnet` | Claude Sonnet |
| `model:haiku` | Claude Haiku |
| `model:gpt-5.4` | GPT 5.4 |
| `model:gpt-5.2-codex` | GPT 5.2 Codex |

## Repo Resolution

When an issue is triggered, the agent resolves the target GitHub repo using a 4-step cascade (highest to lowest priority):

| Priority | Method | Description |
|----------|--------|-------------|
| 1 | **Project → repo mapping** | Static mapping from Linear project IDs |
| 2 | **Team → repo mapping** | Static mapping from Linear team IDs, with optional label filtering |
| 3 | **Linear's `issueRepositorySuggestions` API** | Linear's built-in suggestion (≥ 70% confidence threshold) |
| 4 | **LLM classifier** | Claude Haiku classifies based on issue content, labels, and available repo descriptions |

If the LLM classifier has low confidence, the agent emits an **Elicitation** activity asking the user to clarify which repo to use.

---

## Agent Activity Types

The bot uses Linear's native Agent Activity system:

| Activity | When | What Users See |
|----------|------|----------------|
| **Thought** | Analyzing issue, resolving repo | Thinking indicator in Linear |
| **Response** | Session created, PR opened | Comment-like message on the issue |
| **Error** | Something went wrong | Error message on the issue |
| **Action** | Tool calls (file edits, commands) | Ephemeral status, e.g. `"Editing src/foo.ts"` |
| **Elicitation** | Repo classification is uncertain | Question asking the user to clarify |

## API Endpoints

All `/config/*` endpoints require HMAC auth via `Authorization: Bearer <token>`.

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Health check |
| `/webhook` | POST | Linear webhook receiver |
| `/oauth/authorize` | GET | Start OAuth installation flow |
| `/oauth/callback` | GET | OAuth callback handler |
| `/config/team-repos` | GET/PUT | Team → repo mapping |
| `/config/project-repos` | GET/PUT | Project → repo mapping |
| `/config/user-prefs/:userId` | GET/PUT | Per-user model and reasoning preferences |
| `/config/triggers` | GET/PUT | Trigger configuration (legacy) |
| `/callbacks/complete` | POST | Completion callback from control plane |
| `/callbacks/tool_call` | POST | Tool progress callback from control plane |

## Troubleshooting

<AccordionGroup>
  <Accordion title="Bot not responding to issue assignments">
    1. Verify the webhook URL is correctly configured in your Linear Application settings
    2. Confirm `enable_linear_bot = true` in `terraform.tfvars` and run `terraform apply`
    3. Ensure the OAuth installation was completed by visiting `/oauth/authorize`
    4. Check Cloudflare Worker logs for webhook delivery errors
  </Accordion>
  <Accordion title="Agent can't determine which repo to use">
    The LLM classifier was unable to resolve the repo with sufficient confidence. Either:
    - Add a static team or project → repo mapping (see [Configure Repo Mapping](#4-configure-repo-mapping-optional))
    - Answer the Elicitation question the agent posts on the issue
    - Add a `model:<name>` label or ensure the issue description mentions the repository
  </Accordion>
  <Accordion title="OAuth installation fails">
    The OAuth flow requires admin permissions in the Linear workspace. Ensure the user completing the OAuth flow is a workspace admin. If the callback URL doesn't match, update your Linear Application settings with the correct callback URL.
  </Accordion>
  <Accordion title="Agent activities not appearing in Linear">
    Verify the webhook events **Agent session events**, **Issues**, and **Comments** are all enabled in your Linear Application webhook settings. The webhook URL must be set to `https://<your-bot-worker>/webhook`.
  </Accordion>
</AccordionGroup>
