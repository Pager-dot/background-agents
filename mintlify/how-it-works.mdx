## The Background Model

Unlike interactive coding assistants, Open-Inspect decouples your presence from the work:

```
Traditional:  You type → AI responds → You watch → Repeat

Open-Inspect: You send prompt → Session runs in background → Check results when ready
```

This enables workflows like:

- **Fire and forget** — Notice a bug before bed, kick off a session, review the PR in the morning
- **Parallel sessions** — Run multiple approaches simultaneously without tying up your machine
- **Multiplayer** — Share a session URL with a colleague and collaborate in real-time
- **Unlimited concurrency** — Your laptop isn't the bottleneck

---

## Sessions

A **session** is the core unit of work. Each session is:

- **Tied to a repository** — The agent works in a clone of your repo
- **Persistent** — State survives across connections
- **Multiplayer** — Multiple users can join and see events in real-time
- **Stateful** — Contains messages, events, artifacts, and sandbox state

### Session Lifecycle

```
Created → Active → Archived
            ↑
            └── Restorable
```

### What's Stored in a Session

| Data | Description |
|------|-------------|
| Messages | Prompts sent and their metadata |
| Events | Tool calls, token streams, status updates |
| Artifacts | PRs created, screenshots captured |
| Participants | Users who have joined |
| Sandbox state | Reference to the current sandbox and its snapshot |

Each session gets its own SQLite database in a Cloudflare Durable Object — isolated and high-performance even with hundreds of concurrent sessions.

## Architecture

Open-Inspect uses a three-tier architecture spanning multiple cloud providers:

```
┌──────────────────────────────────────────────────────────┐
│                        Clients                            │
│            ┌──────────┬──────────┬──────────┐            │
│            │   Web    │  Slack   │  GitHub  │            │
│            └────┬─────┴────┬─────┴────┬─────┘            │
└─────────────────┼──────────┼──────────┼───────────────────┘
                  │          │          │
                  ▼          ▼          ▼
┌──────────────────────────────────────────────────────────┐
│                Control Plane (Cloudflare)                 │
│  ┌──────────────────────────────────────────────────┐    │
│  │           Durable Objects (per session)           │    │
│  │  ┌────────┐ ┌──────────┐ ┌────────┐ ┌────────┐  │    │
│  │  │SQLite  │ │WebSocket │ │ Event  │ │Sandbox │  │    │
│  │  │ State  │ │   Hub    │ │Stream  │ │Lifecycle│  │    │
│  │  └────────┘ └──────────┘ └────────┘ └────────┘  │    │
│  └──────────────────────────────────────────────────┘    │
│  ┌──────────────────────────────────────────────────┐    │
│  │            D1 Database (shared state)             │    │
│  │    Sessions index, repo metadata, secrets         │    │
│  └──────────────────────────────────────────────────┘    │
└──────────────────────────┬───────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│                   Data Plane (Modal)                      │
│  ┌──────────────────────────────────────────────────┐    │
│  │                  Session Sandbox                  │    │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐        │    │
│  │  │Supervisor│──│ OpenCode │──│  Bridge  │────────────▶ Control Plane
│  │  └──────────┘  └──────────┘  └──────────┘        │    │
│  │           Full Dev Environment                    │    │
│  │    (Node.js 22, Python 3.12, git, Chrome)         │    │
│  └──────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘
```

### Control Plane (Cloudflare Workers)

The coordinator — it manages state and routes messages without executing code.

**Responsibilities:**
- Session state management (SQLite in Durable Objects)
- WebSocket connections for real-time streaming
- Sandbox lifecycle orchestration (spawn, snapshot, restore)
- GitHub integration (repo listing, PR creation)
- Authentication and access control

### Data Plane (Sandbox Backends)

Where code actually runs. Each session gets an isolated sandbox.

**What's in a sandbox:**
- Debian Linux with common dev tools
- Node.js 22, Python 3.12, git, Bun, GitHub CLI
- `agent-browser` CLI + headless Chrome (browser automation)
- Code-server (browser-based VS Code)
- OpenCode (the coding agent)

Open-Inspect supports two backend patterns:

<Tabs>
  <Tab title="Modal">
    Near-instant startup plus filesystem snapshot restore. Supports repo-image prebuilding and live filesystem snapshots.
  </Tab>
  <Tab title="Daytona">
    Persistent stop/start sandboxes via direct REST API calls. The control plane stops the sandbox on inactivity, then resumes the same sandbox later.

    <Note>
      Daytona does not automatically inject LLM API keys. Add `ANTHROPIC_API_KEY` as a global secret in Settings after deploying.
    </Note>
  </Tab>
</Tabs>

## The Sandbox Lifecycle

### Fresh Start (No Snapshot)

```
Sandbox Created → Git Sync → setup.sh → start.sh → Agent Start → Ready
```

1. Modal spins up a new container from the base image
2. Clones your repository using GitHub App credentials
3. Runs `.openinspect/setup.sh` (if present)
4. Runs `.openinspect/start.sh` (if present)
5. OpenCode server starts and connects back to the control plane

### Restore From Snapshot

```
Restore Snapshot → Quick Git Sync → start.sh → Ready
```

Snapshots include installed dependencies, built artifacts, and workspace state. Follow-up prompts in an existing session start almost instantly.

### Repo Image Start

1. Fast incremental git sync (fetch + hard reset)
2. `setup.sh` is **skipped** — it already ran when the image was built
3. `start.sh` executes for per-session runtime startup
4. Agent starts once the runtime hook succeeds

### When Snapshots Are Taken

- After successful prompt completion
- Before sandbox timeout (inactivity)
- On explicit save triggered by the control plane

### Sandbox Warming

To minimize perceived latency, sandboxes warm proactively as you start typing a prompt — often ready before you hit Enter.

## How Prompts Flow

```
User → Client → Control Plane → Sandbox → OpenCode
                    │                         │
                    │      Events stream back  │
                    ◀─────────────────────────┘
                    │
                    ▼
               Broadcast to all clients
```

1. **You send a prompt** via web or Slack
2. **Control plane queues it** in the session's Durable Object
3. **Sandbox receives the prompt** via WebSocket
4. **OpenCode processes it** — reading files, making edits, running commands
5. **Events stream back** — tool calls, token streams, status updates
6. **Control plane broadcasts** events to all connected clients and stores them in SQLite
7. **Artifacts are created** — PRs and screenshots announced to clients

### Prompt Queuing

If you send a prompt while the agent is working, it's queued:
```
Prompt 1 (processing) → Prompt 2 (queued) → Prompt 3 (queued)
```

## The Agent

Open-Inspect uses [OpenCode](https://opencode.ai) as its coding agent — an open-source agent designed to run as a server.

### Agent Capabilities

| Capability | Description |
|------------|-------------|
| Read files | Explore the codebase, understand context |
| Edit files | Make changes, refactor code |
| Run commands | Execute tests, builds, scripts |
| Git operations | Commit changes, create branches |
| Web browsing | Look up documentation, research errors |
| Visual verification | Use Playwright to check UI changes |

### Commit Attribution

```
Author:    Jane Developer <jane@example.com>
Committer: Open-Inspect <bot@open-inspect.dev>
```

### Pull Request Creation

1. Agent pushes the branch using GitHub App credentials
2. Control plane creates the PR using *your* GitHub OAuth token
3. PR appears as created by you — not a bot — preserving code review workflows

## Real-Time Events

### Event Types

| Event | Description |
|-------|-------------|
| `sandbox_spawning` | Sandbox is being created |
| `sandbox_ready` | Sandbox is ready to accept prompts |
| `sandbox_event` | Tool call, token stream, or other agent event |
| `artifact_created` | PR created, screenshot captured |
| `presence_update` | User joined or left the session |
| `session_status` | Session state changed |

### Multiplayer

Multiple users can connect to the same session simultaneously:
- **Presence** — See who else is watching
- **Shared stream** — Everyone sees the same events
- **Attributed prompts** — Each prompt is tagged with who sent it

## Security Model

Open-Inspect is designed for **single-tenant deployment** where all users are trusted members of the same organization.

### Token Architecture

| Token | Purpose | Scope |
|-------|---------|-------|
| GitHub App Token | Clone repos, push commits | All repos where App is installed |
| User OAuth Token | Create PRs, identify users | Repos the user has access to |
| Sandbox Auth Token | Authenticate sandbox → control plane | Single session |
| WebSocket Token | Authenticate client connections | Single session |

### Secrets

Secrets (API keys, credentials) can be configured at global or per-repository scope:

- **Global secrets** apply to all repositories
- **Repository secrets** apply to a single repo and override global secrets with the same key
- Stored encrypted (AES-256-GCM) in D1
- Never exposed to clients — only key names are visible

### Deployment Recommendations

1. Deploy behind your organization's SSO or VPN
2. Install GitHub App only on repositories you want accessible
3. Use "Select repositories" (not "All repositories") when installing the App

## Repository Lifecycle Scripts

Repositories can define optional startup scripts under `.openinspect/`:

<CodeGroup>
```bash setup.sh (provisioning)
#!/bin/bash
# Runs on fresh sessions and image builds
npm install
pip install -r requirements.txt
```

```bash start.sh (runtime startup)
#!/bin/bash
# Runs on every non-build session startup
docker compose up -d postgres redis
```
</CodeGroup>

| Script | When It Runs | Failure Behavior |
|--------|-------------|------------------|
| `setup.sh` | Fresh sessions + image builds | Non-fatal for fresh sessions; fatal in build mode |
| `start.sh` | Every non-build session startup | Always fatal if present and fails |

Both scripts receive `OPENINSPECT_BOOT_MODE` env var: `build`, `fresh`, `repo_image`, or `snapshot_restore`.

Default timeouts: `SETUP_TIMEOUT_SECONDS=300`, `START_TIMEOUT_SECONDS=120`.
